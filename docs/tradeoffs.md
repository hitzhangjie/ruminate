## 一、Ingest 与 Lint 的职责分离

### 问题

`ruminate ingest` 在写入内容时，是否应该同步检测矛盾（contradiction）和冲突？

### 决策

**不在 ingest 阶段做语义校验，交给 `ruminate lint` 异步完成。**

### 理由

1. **Ingest 保持轻量**：ingest 的核心职责是"把内容快速落盘"——创建/更新 Markdown 文件，更新索引。如果在写入时引入 LLM 调用做矛盾检测，会显著增加延迟，阻塞用户的写入流程。

2. **Lint 可独立迭代**：健康检查规则可以持续丰富——从简单的死链检测、孤立页面发现，到语义层面的矛盾分析——这些都不应影响 ingest 的稳定性和性能。

3. **用户可控的检查节奏**：用户可以一次性 ingest 大量内容，然后统一跑一次 lint 来发现问题，而不是每次写入都被打断。这更符合"快速记录、事后整理"的工作流。

4. **类比编译器与静态分析工具**：编译器只管生成可执行文件，lint/静态分析工具管代码质量。两者职责分离、独立演进，是成熟的工程实践。

### 影响

- Ingest 只做写入，不保证语义一致性
- 用户需要主动执行 `ruminate lint` 来发现 Wiki 中的问题
- Lint 引擎需要设计为可扩展的检查规则框架
