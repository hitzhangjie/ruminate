# Ruminate

> *"The wise man knows that it is better to sit on the bank of a remote mountain stream than to be emperor of the whole world."* — but between the two of us, the real wise move is to write it all down.

**Ruminate**（反刍）是一个 AI 驱动的个人知识库系统。它不只是存储你的笔记，而是帮你持续**消化**信息——收集、整理、关联、更新、淘汰——让你的知识像活的有机体一样不断生长和进化。

---

## 为什么需要 Ruminate？

### 好记性不如烂笔头

我们都听过这句话，但真正的问题是：**烂笔头写下来之后呢？**

大多数人陷入了两种困境：

1. **写而不理**：笔记越积越多，文件夹越来越深，但从来不去回顾。知识变成了数字尘埃。
2. **理而不及**：你确实在维护笔记——更新博客、整理错题本、修订电子书——但维护成本随着知识量增长而指数级上升。更新一个观点需要追溯几十处引用，发现一条矛盾需要翻遍所有相关文档。

**好学生喂不饱，坏学生撑死。** 这不仅仅是学习能力的问题，更是知识管理方法的差异。善于学习的人，他们的知识是活的——不断被质疑、修正、关联、强化。而不善于学习的人，他们的知识是死的——堆积在那里，既不消化也不排出。

Ruminate 要解决的问题是：**让每个人都能科学地维护自己头脑中的知识库。** 让好学生变得更好，让差生也能掌握更科学地成长路径。

### 从 RAG 到 Wiki：知识需要积累

现有的 AI 知识工具（NotebookLM、ChatGPT 文件上传、各类 RAG 系统）有一个共同缺陷：**每次查询都是从零开始。**

你上传了 50 篇论文，问了一个需要综合 5 篇论文才能回答的问题。AI 临时检索、临时拼凑、临时推理——然后答案消失在聊天记录里。下次你问一个相关但更深的问题，它又重新来过。**没有任何积累。**

Ruminate 的核心思想不同：**AI 增量构建并持续维护一个持久化的知识 Wiki**——一个结构化的、相互链接的 Markdown 文件集合，它位于你和原始资料之间。

当你添加一份新资料时，AI 不只是索引它。它读它、提取关键信息、整合进已有 Wiki——更新实体页面、修订主题摘要、标注新旧矛盾、强化或质疑当前的综合结论。**知识被编译一次，然后持续保鲜。**

这就是关键区别：**Wiki 是一个持久化的、复合增长的产物。** 交叉引用已经就位，矛盾已经标注，综合结论已经反映你读过的所有内容。每添加一份资料、每提出一个问题，Wiki 都会变得更丰富。

## Ruminate 是什么

Ruminate 是一个**完整的知识库软件系统**，而不是一套需要小心调教的 AI prompt 或 skill。它开箱即用，交付给任何用户都能稳定运行。

### 三层架构

```
┌──────────────────────────────────────────┐
│              原始资料层                    │
│  文章、论文、图片、数据文件……               │
│  只读 — 永远不被修改，是你的事实来源         │
└──────────────────────────────────────────┘
                    ↓ AI 读取
┌──────────────────────────────────────────┐
│              知识 Wiki 层                  │
│  摘要、实体页、概念页、对比、综述、索引……     │
│  完全由 AI 生成和维护，你只管阅读和提问       │
└──────────────────────────────────────────┘
                    ↓ 受控于
┌──────────────────────────────────────────┐
│              Schema 配置层                 │
│  定义 Wiki 结构、命名规范、工作流、输出格式…  │
│  你和 AI 共同演化，越用越好                  │
└──────────────────────────────────────────┘
```

### 核心操作

| 操作 | 描述 |
|------|------|
| **摄入 (Ingest)** | 放入新资料 → AI 自动阅读、提取、整合，一份资料可能触及 10-15 个 Wiki 页面 |
| **查询 (Query)** | 基于 Wiki 提问 → AI 搜索相关页面，综合回答并附带引用。好的回答可以**回写进 Wiki**，不让它消失在聊天记录里 |
| **巡检 (Lint)** | 定期健康检查 → 发现矛盾、过时内容、孤立页面、缺失的交叉引用、可以深入的方向 |

### 关键能力

- **多源接入**：通过 MCP 协议连接不同信息源（网页剪辑、RSS、播客转录、Slack、会议记录……），统一汇入知识库
- **完全重建**：所有资料可以随时从头重建整个 Wiki，不被历史累积的偏差所困
- **增量更新**：新增资料只触发相关的增量修改，不必推倒重来
- **Git 原生**：Wiki 就是 Markdown 文件的 Git 仓库，自带版本历史、分支和协作能力
- **本地优先**：你的知识完全在你自己的磁盘上，不依赖任何云服务

## 与卡帕西 (Andrej Karpathy) 的共鸣

我经常写总结、写博客、写电子书，使用的编辑器、文件格式也不完全统一，如何维护这些碎片化的知识，是我经常思考的一个问题：
- 在LLM大爆发之前：我总是想找类似 notion/docmost/appflowy/affine 之类的功能强大、可插件化集成的编辑器，而且希望它们中有一个能够跑出来成为我的主力工具，但这种想法是种”理想主义”，没有一个软件厂商可以满足我的全部需求；
- 在LLM大爆发之后：我意识到应该基于AI构建一个更强大的知识库入口，它可以根据我过往、当下偏好的碎片化知识管理的方式，从这些不同的信息源进行收集并重复打磨、咀嚼、更新这些知识，让知识时刻保持新鲜并富有活力。

本项目受到 [Karpathy 的 LLM Wiki](llm_wiki.md) 理念的直接启发，他在其中提出了完全相同的核心洞察：

> *"The tedious part of maintaining a knowledge base is not the reading or the thinking — it's the bookkeeping. LLMs don't get bored, don't forget to update a cross-reference, and can touch 15 files in one pass."*

维护知识库最痛苦的不是阅读或思考，而是**记账**——更新交叉引用、保持摘要时效、标注新旧矛盾、在几十个页面之间保持一致性。人类放弃 Wiki 是因为维护负担的增长速度超过了价值增长速度。LLM 不会厌倦、不会忘记更新引用、一次操作可以触及 15 个文件。

Ruminate 将这个理念从一个"idea file"变成了一个**可交付的软件系统**。
